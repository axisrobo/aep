# Harmovela 愿景

## 一句话愿景

Harmovela 为自主系统提供开放的协调层，使其能够共同感知、决策、行动、恢复和进化。

## 问题

现代自主系统面临的协调挑战远远超出同步工具调用的范畴。智能体、工具、记忆系统和运行时必须在多个相互依存的维度上进行协调，而共享协议的缺失迫使每个系统发明自己的模式。

### 超越事件和异步

事件流和异步通信是必要的，但并不充分。自主系统还需要以下方面的标准原语：

- **状态管理**：智能体必须知道当前什么是真的、何时状态变得陈旧、何时发生变化——而无需每个消费者轮询或猜测。
- **委派**：工作必须通过分配、接受、交接、升级和取消在实体间流动——不是没有生命周期保证的临时消息传递。
- **恢复**：当组件故障、重启或丢失连接时，系统必须通过幂等、重放、检查点和补偿来恢复——不是每个集成中自定义的重试逻辑。
- **治理**：协调跨越信任边界——身份、授权、审计、租户隔离——需要可验证的协议级边界，而非应用级约定。

没有统一的协调协议，每个维度都会在每个厂商、每个运行时和每个集成中反复重新发明。轮询循环、自定义 webhook、工具特定回调、临时消息队列和 prompt 级约定大量扩散。这些方法无法在智能体运行时、工具、记忆系统和部署环境之间组合。

## 愿景

Harmovela 将七个相互依存的协调维度整合进一个开放协议：

| 维度 | Harmovela 的定义 |
|---|---|
| 事件 | 发生了什么 — 发布、订阅、关联、重放、确认。 |
| 任务 | 正在执行什么 — 从提交到完成、失败或取消的生命周期。 |
| 状态 | 当前事实 — 版本化状态、新鲜度窗口、失效、变更传播。 |
| 上下文/记忆 | 认知输入 — 更新、失效、溯源、检索就绪通知。 |
| 委派 | 谁做什么 — 分配、接受、交接、升级、取消传播。 |
| 恢复 | 故障应对 — 幂等、重放、检查点、中断、补偿。 |
| 治理 | 谁能做什么 — 身份、授权、审计、租户隔离、策略集成。 |

这些维度不是孤立的功能。任务生命周期产生事件。委派链产生恢复义务。状态变化触发上下文失效。治理约束所有其他维度。协议将它们视为一个统一的整体，而不是功能的集合。

这种统一产生了三种相互交织的品质：

- **连接与协作**：自主实体间的网络效应、随参与者增长而增强的集体智能、智能体与工具及运行时之间的对齐。
- **动态与进化**：对变化环境的持续适应、协调行为的涌现、上下文与状态随时间的流动。
- **秩序与治理**：通过可验证边界建立的信任、自主与约束之间的平衡、不确定性下的韧性协调。

## 核心用例

### 事件流

智能体需要从工具、环境、记忆系统、编排器和其他智能体接收事件。

示例：

- 记忆系统发出 `memory.fact.invalidated`
- 浏览器观察器发出 `environment.page.changed`
- 机器人传感器发出 `environment.object.detected`
- 工作流引擎发出 `task.blocked`

Harmovela 提供发布、订阅、关联、重放和确认语义，使事件成为一等协议对象，而非临时 JSON 消息。

### 任务生命周期

有些工具调用太慢或太开放，不适合同步 RPC。Harmovela 将这些表示为具有完整生命周期的任务：已提交、已接受、执行中、推进中、已完成、已失败、已取消和已超时。每个转换都是消费者可以订阅的事件。

### 状态与上下文

智能体需要在已知信息变得陈旧时做出反应。Harmovela 定义版本化状态更新、新鲜度窗口、失效和变更传播，使消费者收到结构化通知而非轮询。

### 委派与恢复

智能体需要跨实体共享工作。Harmovela 将分配、接受、交接、升级和取消传播定义为协议级原语。当组件发生故障时，协议通过幂等、重放、检查点和补偿来规定恢复——而非自定义重试逻辑。

### 治理

自主协调需要可验证的边界。Harmovela 在协议级别提供身份、授权、审计和租户隔离，使信任边界是显式的且可机器验证的——而非应用级约定。

## 目标

- 定义一个统一的协议级协调模型，整合事件、任务、状态、上下文/记忆、委派、恢复和治理维度。
- 为任务生命周期、工具反馈、上下文变化、记忆更新、智能体消息和委派流提供标准事件类型。
- 通过发布/订阅、委派和共享上下文实现连接与协作——在自主实体间增长网络效应。
- 通过版本化状态、新鲜度窗口、失效和变更传播支持动态适配。
- 通过身份、授权、审计、租户隔离和策略集成提供秩序与治理。
- 支持跨多种传输的本地和分布式运行时。
- 在对话、任务、工具调用和事件之间保持因果关系和关联。
- 支持断连后的重放和恢复。
- 保持核心足够小，以便轻松实现，同时保持领域特定需求的可扩展性。

## 非目标

- Harmovela 不是 MCP 的替代品。
- Harmovela 不是新的 LLM 推理 API。
- Harmovela 不是通用数据库 API。
- Harmovela 不是强制性的消息代理实现。
- Harmovela 不定义每个领域特定的事件模式。
- Harmovela 不定义通用智能模型或普适本体。

## 设计原则

- **连接先于控制**：在约束行为之前先跨边界协调——开放性实现网络效应。
- **协议先于平台**：在选择基础设施之前先定义可互操作的消息。
- **兼容 MCP，不依赖 MCP**：与 MCP 集成，同时在 MCP 之外仍然有用。
- **事件优先**：异步通信不应被建模为虚假的同步调用。
- **适配是显式的**：状态新鲜度、失效和变更传播必须被声明，而非推断。
- **信任边界即协议边界**：身份、授权和审计必须嵌入协议中，而非推迟到应用程序逻辑。
- **持久性可选，语义明确**：传输可能短暂的或持久的，但交付保证必须被声明。
- **因果关系至关重要**：每个事件应该在可用时追溯到任务、对话或触发事件。
- **人类可读，机器可验证**：JSON 应易于检查和模式校验。
- **核心小，领域可扩展**：标准化信封和生命周期事件；允许领域特定的负载。
