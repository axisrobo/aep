# AEP 愿景

## 一句话愿景

AEP 让异步通信成为 LLM 智能体的一等公民能力。

## 问题

现代智能体系统不再是简单的请求-响应程序。它们在后台运行任务，观察外部环境，更新记忆，与其他智能体协作，接收延迟的工具结果，并在原始提示已经结束之后对上下文变化做出反应。

MCP 标准化了智能体如何同步地发现和调用工具。这是必要的，但还不够。智能体还需要一种标准的方式来接收事件、进度、状态变化、记忆更新和延迟结果。

没有这样的协议，每个运行时就不得不发明自己的模式：

- 轮询任务状态
- 自定义 webhook
- 工具特定的回调
- 临时消息队列
- 非标准的 JSON 事件
- 不可被机器验证的 prompt 级别约定

这些方法无法在不同的智能体运行时、工具、记忆系统和部署环境之间良好地组合。

## 愿景

AEP 应成为智能体系统的通用异步事件层。

一个智能体应该能够订阅记忆流、启动长时间运行的工具任务、接收进度事件、观察外部状态变化、取消工作、重放遗漏事件，并将所有这些与一次对话或一个任务关联，而无需依赖某个特定厂商的运行时。

## 核心用例

### 事件流

智能体需要从工具、环境、记忆系统、编排器和其他智能体接收事件。

示例：

- 记忆系统发出 `memory.fact.invalidated`
- 浏览器观察器发出 `environment.page.changed`
- 机器人传感器发出 `environment.object.detected`
- 工作流引擎发出 `task.blocked`

### 异步工具调用

有些工具调用太慢或太开放，不适合同步 RPC。

示例：

- Web 爬取
- 长时间的研究任务
- 代码库索引
- 视频或音频处理
- 模拟
- 多步骤规划

AEP 应将这些表示为任务，使用 accepted、progress、output、completed、failed、cancelled、timed-out 事件。

### 上下文和记忆事件

智能体需要在上下文变化时做出反应。

示例：

- 新的相关记忆已被索引
- 之前可信的事实被宣告无效
- 约束发生变化
- 用户偏好被更新
- 检索到的上下文包已就绪

### 智能体间通信

智能体需要在彼此之间交换消息而不阻塞对方。

示例：

- 研究员智能体向规划器发布发现结果
- 审查员智能体标记一个风险
- 工作智能体请求澄清
- 协调器取消重复工作

## 目标

- 定义一个适合 LLM 智能体的协议级事件信封
- 为任务生命周期、工具反馈、上下文变化、记忆更新和智能体消息提供标准事件类型
- 支持本地和分布式运行时
- 支持多种传输方式，而不让协议依赖于某个特定的消息代理
- 保持跨对话、任务、工具调用和事件的因果关系和关联
- 支持断连后的重放和恢复
- 保持核心足够小，以便轻松实现

## 非目标

- AEP 不是 MCP 的替代品
- AEP 不是新的 LLM 推理 API
- AEP 不是通用数据库 API
- AEP 不是强制性的消息代理实现
- AEP 不定义每个领域特定的事件模式

## 设计原则

- **协议先于平台**：在选择基础设施之前先定义可互操作的消息
- **兼容 MCP，不依赖 MCP**：与 MCP 集成，同时在 MCP 之外仍然有用
- **事件优先**：异步通信不应被建模为虚假的同步调用
- **持久性可选，语义明确**：传输可能短暂或持久的，但交付保证必须被声明
- **因果关系至关重要**：每个事件应该可以追溯到任务、对话或触发事件（当可用时）
- **人类可读，机器可验证**：JSON 应易于检查和模式校验
- **核心小，领域可扩展**：标准化信封和生命周期事件；允许领域特定的负载
