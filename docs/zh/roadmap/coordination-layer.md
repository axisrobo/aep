# 协调层 — 活跃前沿

> 属于 [Harmovela 路线图](../roadmap.md)。本层回答 **"谁做什么，基于什么共享事实？"** —— 让多个自主实体在[事件层](event-layer.md)之上协同工作的语义。

**自主能力映射：** L1（有界自主任务智能体）→ L2（多智能体协作与委派）。

**状态：** 活跃推进。原语已存在；语义需收紧后才能跨独立实现安全互操作。**全部 10 个维度模块已在 TypeScript、Python、Go 和 Java 中存在，适配器连接已完成。** 证据见 `docs/protocol/compatibility-matrix.md`。

## 目的

事件告知每个参与方发生了什么。协调告知参与方如何协同行动：谁拥有任务、工作如何交接、冲突状态如何调和、取消如何沿委派链传播。这是协议当前的前沿 —— 亦即使 Harmovela 超越消息总线的层。

## 为何这是当前的关键缺口

自主能力阶梯将两个级别置于该层：

- **L1 — 有界自主任务智能体。** 当前已支持，但正确行为仍依赖实现者提供的策略。Harmovela 承载任务生命周期，但尚未文档化实现者为保持任务有界而必须提供的策略表面。
- **L2 — 多智能体协作与委派。** 仅部分支持。委派、交接、升级和取消传播事件已存在，但其语义松散到两个独立实现可能对其有不同解读。

填补这一缺口是 0.2–0.4 发布的中心目标。

全部 10 个 Harmovela 协调维度（事件、恢复、治理、任务、状态、上下文/记忆、委派、工具、智能体、环境）已提取为独立模块，覆盖全部四种语言实现并具备完整的适配器连接。此外，5 个基础设施模块（harness、conformance、CLI、runtime 和 MCP bridge）提供跨领域的构建、测试、传输和工具集成表面。事件类型分类记录在 `docs/protocol/event-dimension-classification.md` 中。维度迁移证据记录在 `docs/protocol/compatibility-matrix.md` 中。

在事件/治理合约基础通过其共享 fixtures 且兼容性矩阵对每个受影响的公共表面有批准的决策之前，任何协调维度不得从旧有实现中提取。

## 工作项

### 1. 文档化 L1 策略表面

有界自主能力只有在边界被明确指定时才是真实的。L1 中，预算是一项咨询性的、由实现者提供的声明（例如成本、时间或操作次数）。它可以约束某运行时的行为，但 Harmovela 尚未定义可移植的强制执行、授权、违规事件或其一致性。这些成为 L3 适配层的要求。文档化超时策略、允许的操作以及终止条件，使 L1 智能体通过合约而非期望来被约束。以文档化和协商能力的形式交付，不涉及新的线缆行为。

**已交付：** [`docs/protocol/l1-policy-surface.md`](../protocol/l1-policy-surface.md) —— 咨询性策略表面，涵盖预算限制、超时策略、允许的操作、终止条件和协商。与 L3 强制执行向前兼容。

### 2. 收紧委派语义

使分配、接受、交接、升级和取消传播无歧义：

- 同一时间仅有一个所有者；所有权转移是一个可观察、有序的事件。
- 父任务的取消确定性地传播到被委派的子任务。
- 升级具有定义的目标选择和确认合约。
- 被拒绝或超时的交接具有定义的回落行为。

每条规则均配备正向和负向一致性 fixtures。

### 3. 强化共享事实（状态）一致性

协调要求对当前事实达成共识。收紧版本化状态更新、新鲜度窗口、失效和变更传播，使消费者无需轮询即可收敛到相同视图，并使过期状态冲突具有定义的解决方式。

### 4. 将协调打包为可采纳的配置文件

将稳定核心与可独立采纳的协调能力分离。委派配置文件必须定义其标识符、依赖、能力协商、版本控制和一致性 fixtures，使实现可以显式声明 L2 支持。

## 发布门禁

### 0.2 Core Stabilization

进入标准：
- 公开的 Harmovela 标识和旧有技术标识符的兼容性策略。

退出标准：
- 冻结的 L0-L1 核心：信封、会话、订阅、任务生命周期、错误、关联、版本协商和声明的交付语义。
- 共享的正向和负向生命周期 fixtures 覆盖接受、进展、阻塞工作、终止结果、取消和无效转换。
- 一致性等级、manifest 期望和默认运行器目标内部一致。
- L1 咨询性策略表面已文档化且可进行能力协商。

### 0.3 Optional Profiles

进入标准：
- 0.2 核心兼容性策略和生命周期 fixtures 已发布。

退出标准：
- 规范的 `harmovela.coordination.v1` L2 配置文件定义任务、状态和委派范围、标识符、依赖、能力协商、版本控制和一致性 fixtures。
- `harmovela.runtime-semantics.v1` 保留信念、新鲜度、中断、补偿和溯源的语义。
- 委派、所有权转移、交接、升级和取消传播具有正向和负向 fixtures。
- 实现可声明和按配置文件筛选一致性。
- 安全配置文件 fixtures 覆盖安全配置文件已定义的授权和租户隔离行为；审计在 L3 策略/审计工作之前保持实现特定。

### 0.4 Beta

进入标准：
- 0.3 配置文件声明和 fixtures 已完整。
- 冻结核心具有已发布的兼容性策略。

退出标准：
- 命名的 `harmovela.coordination.v1` manifest/profile 声明已发布。
- 至少两个独立维护的实现运行相同的已声明、已命名、已版本化的标准拓扑（`stdio` 或 `WebSocket`）于 L2 协调配置文件。
- 异步任务编排、共享状态/上下文失效和委派任务交接场景各自通过其声明的 fixture 断言。
- 公开一致性矩阵记录实现版本、配置文件、拓扑标识符和版本，以及版本化的场景和 fixture 标识符和结果。
- 零个核心或 L2 一致性问题被分类为 blocker 或 critical。
- 社区治理提案已发布。

协调可互操作且稳定后，工作将移向[适配层](adaptation-layer.md)。
