# 事件层 — 基础

> 属于 [Harmovela 路线图](../roadmap.md)。本层回答 **"发生了什么？"** —— 所有其他内容所依赖的类型化、可关联的通信基板。

**自主能力映射：** L0（事件感知智能体）。

**状态：** 已交付且稳定。此为协议的冻结基线。

## 目的

事件层是通信基板。它定义了自主实体如何发布、订阅、关联、重放和确认类型化事件。在此层稳定之前，其上的一切内容都无意义：协调和适配都假设存在可靠的事件基板。

## 本层提供的内容

- 具有稳定身份标识、来源、目标、因果、关联和时间戳字段的通用事件信封。
- 会话初始化和能力协商。
- 按主题、类型模式、来源、目标、对话、任务或域的订阅。
- 作为事件流的任务生命周期（提交→完成、失败、取消、超时）。
- 具有声明式交付语义的多种传输绑定。
- 持久化交付、重放游标、确认、重试和死信原语。
- 使用共享 fixtures 的跨语言一致性。

## 已交付工作（Phase 0–8）

事件层是项目前八个交付阶段的累积成果。全部已完成。

| Phase | 对事件层的贡献 |
| --- | --- |
| 0. 愿景与设计 | 愿景、架构、协议设计草案、MCP 关系、术语 |
| 1. 核心规范草案 | 信封 schema（`schemas/aep-envelope.schema.json`）、事件类型注册表、会话、订阅、任务生命周期、错误模型、版本控制、共享 fixtures |
| 2. 传输绑定 | stdio、WebSocket、HTTP SSE、gRPC 流式传输，以及 NATS、Kafka 和 Redis Streams |
| 3. 参考实现 | 参考服务器/客户端、JSON Schema 校验、本地路由器、跨语言一致性 |
| 4. MCP 桥接 | MCP 工具调用到任务桥接、MCP 工具的任务完成事件、互操作示例 |
| 5. 可靠性与生产语义 | 重放游标、确认协议、死信事件和重试策略 |
| 6. 生态与一致性 | 公开规范站点、兼容性测试套件、一致性等级、贡献指南和行为准则 |
| 7. 事件族一致性 | 差异文档、六个新事件族和跨语言 fixtures |
| 8. 交付端到端一致性 | `HARMOVELA-C3` 级别含 `delivery_e2e`，四种语言均含完整 DeliveryTracker 追踪、载荷 schema 校验 |

身份标识、授权、审计、租户隔离和注册表治理属于跨领域治理基础。委派和运行时语义记录在协调层中，因其在事件基板之上建立多智能体行为。

## 当前状态

协议涵盖 17 个规范、4 个一致性等级（C0–C3）、7 个跨语言 fixtures、4 种传输绑定、四种语言约 370 个测试用例，以及发布于 https://axisrobo.github.io/harmovela/ 的规范站点。默认一致性目标为 `HARMOVELA-C3`。每种语言均可使用 DeliveryTracker 集成。网络化 PostgreSQL 交付存储后端与内存和 SQLite 后端一同在所有四种语言中提供。

## 剩余基础工作

事件层功能上已完整。唯一待办事项是身份标识，跟踪为发布路径 **0.1 Transition**：

- 在所有文档中将 **Harmovela** 确立为公开协议标识。
- 为旧有技术标识符（schema URI、线缆版本字段、包名、`AEP-C*` 级别）制定明确的兼容性策略。
- 文档化协议标识、schema 标识符、包名和仓库路径的迁移顺序 —— 不静默改变线缆行为或交付保证。

旧有技术标识符（如 `schemas/aep-envelope.schema.json` 和 `AEP-C3`）在版本化迁移完成前保持有效。

## 退出标准

- 公开文档一致性地将 Harmovela 标识为协议。
- 旧有技术标识符具有明确的版本化兼容性策略。
- 冻结的事件基板无回归。

一旦身份标识迁移文档化，事件层的后续工作即告完成，工作重心将转向[协调层](coordination-layer.md)。
