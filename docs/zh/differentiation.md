# Harmovela 与现有事件系统的区别

> 状态：非规范性定位文档。Harmovela 是协议草案；本文档不添加协议要求。

## 范围与诚实的基线

Harmovela 旨在使智能体系统中的异步变更可互操作。它建立在已有的事件和消息传递实践之上，而非取代它们。

- **CloudEvents** 提供了广泛有用的事件元数据模型和可互操作的事件格式映射。
- **AsyncAPI** 可以描述异步 API 操作、通道、消息和传输绑定。
- **Kafka、NATS、Redis Streams** 等消息代理提供持久化或瞬时的传输、路由、保留、消费者协调以及代理特定的交付行为。
- **Actor 运行时** 提供了有用的执行、隔离、监督、邮箱和消息传递模型。
- **事件溯源 (Event Sourcing)** 提供了从仅追加的事件历史中持久化和重建状态的有用方式。

Harmovela 不声称拥有新的通用事件信封、代理、交付算法或传输。这些是已建立的基础设施关注点。相反，Harmovela 注册了一小组与智能体相关的事件名称，并在上述基础设施之上提出关联、生命周期和订阅约定。参见草案 [协议设计](protocol-design.md)、[交付语义](../protocol/delivery.md) 和 [任务生命周期](../protocol/task-lifecycle.md)。

## Harmovela 标准化了什么

区分在于一个概念是全新的、是被一致命名的推荐信封字段，还是由基础设施提供：

| 关注点 | Harmovela 的定位 |
| --- | --- |
| 通用事件元数据和序列化 | 已建立的实践。Harmovela 为其自身事件定义了一个草案信封，但不声称通用信封的新颖性。 |
| `session_id`、`conversation_id` 和 `task_id` | 推荐的 Harmovela 信封字段。应用程序此前已建模过等效概念；预期价值在于对智能体事件使用一致命名的、可互操作的字段，而非强制放置。 |
| `correlation_id` 和 `causation_id` | 已建立的分布式系统概念和推荐的 Harmovela 信封字段，用于操作分组和直接事件溯源。 |
| 任务生命周期事件名称和有效转换 | 智能体级别的协议语义。任务生命周期为生产者和消费者验证观察到的任务流定义了共享基础。 |
| `context.invalidated` 和 `memory.fact.invalidated` | 已注册的 Harmovela 事件类型。其详细的负载模式和消费者语义仍在未来的规范工作中。 |
| 过滤、确认、重放、保留、路由和交付实现 | 共享的协议概念，由传输层具体实现。Harmovela 不取代代理能力，也不做出超越其草案交付规范的更强保证。 |

### 智能体相关的关联

`session_id`、`conversation_id` 和 `task_id` 是智能体运行时常用的三种不同的、可组合的作用域：

- `session_id` 标识一个运行时连接或执行会话。
- `conversation_id` 标识一个交互线程。
- `task_id` 标识一个异步工作单元。

这并非声称此前没有应用程序建模过会话、对话或任务。这些一致命名的字段为独立开发的智能体、工具、上下文提供者和记忆系统提供了一种可互操作的方式，将异步事件附加到相关的运行时、交互或工作作用域。一个事件可以携带这些推荐字段的任意组合；消费者不得仅凭其存在就推断包含或父子关系。`correlation_id` 可以分组更广泛的逻辑操作，而 `causation_id` 则标识直接导致另一个事件的事件。

### 失效是消费者信号

草案注册了 `context.invalidated` 和 `memory.fact.invalidated`，但尚未定义完整的负载模式或规范性的消费者行为。预期的应用层定位是：先前依赖过时上下文或记忆事实的消费者可以评估是否刷新、丢弃、注释或以其他方式修正其工作状态。这些事件不声明一个通用的真值维护系统，不证明每个缓存结论都是错误的，也不保证任何消费者会修正其信念。详细的失效语义仍在未来的规范工作中。

## 对比

以下对比描述的是预期角色，而非某个系统包含另一个系统的主张。

| 能力 | Harmovela | CloudEvents | AsyncAPI | Kafka、NATS、Redis Streams |
| --- | --- | --- | --- | --- |
| 会话、对话和任务关联 | 推荐的、一致命名的信封字段：`session_id`、`conversation_id`、`task_id`、`correlation_id` 和 `causation_id`。同时出现不意味包含关系。 | 通用扩展属性可以携带这些值；其智能体含义不由 CloudEvents 定义。 | 可以文档化包含这些字段的消息，但不定义其运行时语义。 | 头部、键、主题、流和负载可以携带它们；语义由应用程序定义。 |
| 任务生命周期验证 | 任务流的草案共享事件族和转换模型。 | 表示生命周期事件但不定义任务状态或转换。 | 文档化生命周期消息和通道但不定义转换有效性。 | 传输事件；验证属于应用程序或消费者。 |
| 上下文和记忆失效 | 注册 `context.invalidated` 和 `memory.fact.invalidated`；负载模式和规范性消费者行为仍在未来工作中。 | 可以携带失效事件而不定义其含义。 | 可以描述失效消息而不定义消费者行为。 | 传输失效消息而不解释其含义。 |
| 因果关系 | `causation_id` 是推荐的信封字段，用于直接事件溯源。 | 扩展属性可以表示溯源。 | 可以描述一个溯源字段。 | 由应用程序选择的元数据或负载约定。 |
| 订阅过滤 | 草案过滤器包括事件类型、来源、目标、主题、会话、对话、任务、时间和游标。 | 无订阅协议。 | 描述订阅操作；不提供代理。 | 原生过滤因代理和绑定而异。 |
| 重放 | 基于游标的草案重放和确认语义，由具备能力的传输层实现。 | 无重放语义。 | 可以文档化支持重放的绑定。 | 支持因产品、保留策略和消费者配置而异。 |
| 同步能力调用 | 不在 Harmovela 的主要职责内；与 MCP 组合使用。 | 非调用协议。 | 可以描述请求-响应模式，但不是运行时调用层。 | 请求-响应可能可用，但能力发现和调用语义由应用程序定义。 |

| --- | --- | --- | --- | --- |
| 会话、对话和任务关联 | 推荐的、一致命名的信封字段，用于不同的智能体作用域；同时出现不意味包含关系。 | Actor 标识和消息元数据可以建模关系；无通用智能体关联约定。 | 聚合和流标识可以关联事件；智能体作用域关系由应用程序特定定义。 | 会话和请求标识符在可用时存在；任务级异步关系语义不在其核心调用模型内。 |

| 因果关系 | 推荐的 `causation_id` 信封字段。 | 可以在消息元数据或负载中建模。 | 可以在事件负载中建模。 | 请求 ID 可以将响应关联到请求；跨异步图的事件因果关系不是其主要模型。 |

| 同步能力调用 | 委托给 MCP 或其他调用协议。 | 消息 ask 和 future 可以支持调用，但是运行时特定的。 | 事件溯源记录领域事件，不定义能力调用或请求-响应。 | MCP 的主要职责是能力发现和同步调用。 |

## 组合边界

Harmovela 设计为与现有层组合使用，而非替代它们：

- **消息代理** 传输 Harmovela 事件，并提供自身的路由、持久化、排序和消费者机制。
- **CloudEvents 兼容映射** 在 Harmovela 部署需要 CloudEvents 格式或协议绑定时是可行的。此类映射必须保留其使用的 Harmovela 字段和事件语义。
- **AsyncAPI** 可以描述 Harmovela 通道、消息模式和传输绑定。它是一种描述机制，而非事件运行时。
- **MCP** 仍然是能力发现、工具调用、资源读取和提示检索的同步调用层。Harmovela 传递随时间的变化和生命周期事件；它不应成为第二个同步工具协议。参见 [Harmovela 与 MCP](mcp-relationship.md)。
- **事件溯源系统** 可以存储 Harmovela 事件，但 Harmovela 不要求事件溯源或规定如何重建状态。
- **Actor 运行时** 可以发送和消费 Harmovela 事件，但 Harmovela 不要求 Actor 或采用特定的 Actor 监督或邮箱模型。

## 当前局限

草案语义有意止步于多个智能体运行时关注点之前。这些是未来规范工作的领域，而非当前协议保证：

- **信念修正**：失效信号可以触发本地修正，但 Harmovela 不定义一个通用的信念模型或协调算法。
- **委派**：任务事件可以描述工作，但委派权限、所有权转移和子任务问责需要进一步定义。
- **溯源**：`causation_id` 仅提供直接事件溯源；证据、来源信任和推导链需要进一步定义。
- **新鲜度**：时间戳、过期和失效可以辅助新鲜度决策，但 Harmovela 不定义通用的新鲜度策略。
- **中断**：取消事件存在于草案任务生命周期中，但中断安全、回滚、部分效果和跨运行时保证需要进一步定义。

实现应明确说明其传输、保留、排序、重试、授权和消费者状态策略。它们不应在草案 [交付语义](../protocol/delivery.md) 或其部署的具体代理和绑定之外推断保证。
